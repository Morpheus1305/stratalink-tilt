# Phase 1: Add Venue Relays - Aerodrome (Base), Velodrome (Optimism), PancakeSwap (BNB)

## Context
These three venues directly address chain coverage requested by ADGM/FSRA.
Brian Byagaba stated: "Those chains are Ethereum, BNB, typically EVM
compatible chains like Base. So that's the L2s." These are the immediate
priority venues - the dominant DEX on each of the chains he named.

All three are AMM-based DEXs. The relay pattern is identical to the
existing Uniswap (uniswap-relay.ts) and Curve (curve-relay.ts) relays.
Aerodrome and Velodrome share the same codebase (same team, same
contracts), so the second relay should be minimal incremental work after
the first.

## Implementation Order
1. Aerodrome (Base) - build the AMM relay pattern for ve(3,3) pools
2. Velodrome (Optimism) - clone Aerodrome relay, change RPC endpoint
3. PancakeSwap (BNB) - Uniswap V3 fork, reuse existing Uniswap logic

## Venue 1: Aerodrome (Base Chain)

### Overview
- Dominant DEX on Base (Coinbase's Ethereum L2)
- 60%+ of all DEX volume on Base
- $520M+ in 24-hour volume, $1.3B TVL
- AMM model: concentrated liquidity (Slipstream) + classic vAMM/sAMM
- Venue type: amm

### Data Access (in order of preference)

**Option A: Expand Network REST API (recommended)**
- GET /dex/getpoolliquidity - total pool liquidity
- GET /dex/gettokenliquidity - per-token liquidity
- GET /dex/getpool - pool address, tick spacing, reserves
- DexId for Aerodrome: 3600
- Docs: https://docs.expand.network/integrations/dex-and-dex-aggregators/aerodrome

Example:
```
GET https://api.expand.network/dex/getpool
  ?dexId=3600
  &tokenA=0x4200000000000000000000000000000000000006
  &tokenB=0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913
```

**Option B: Bitquery GraphQL API**
- Real-time trades, pool liquidity, OHLCV
- Docs: https://docs.bitquery.io/docs/blockchain/Base/aerodrome-base-api/

**Option C: Direct on-chain reads via Base RPC**
- Pool factory: 0x420DD381b31aEf6683db6B902084cB0FFECe40Da
- Read pool contracts directly

### Relay Implementation
Create: server/services/aerodrome-relay.ts

1. **Pool discovery**: For each ILU-20 token, find the primary liquidity
   pool (TOKEN/USDC or TOKEN/WETH with highest liquidity).
2. **Depth derivation**: Calculate how much token can be bought/sold
   before moving price 10bps from current spot. Same calculation as
   existing Uniswap/Curve relays.
3. **Spread derivation**: For Slipstream (concentrated liquidity) pools,
   derive from tick spacing and liquidity density. For classic pools,
   derive from fee tier and pool depth.
4. **Normalisation output**:
   ```typescript
   {
     venue: "aerodrome",
     venueType: "amm",
     chain: "base",
     depth: number,     // USD at +/-10bps
     spread: number,    // bps
     lastEvent: number, // timestamp
     lisRef: `lis-aerodrome-${timestamp}`
   }
   ```
5. **Polling**: Every 5 seconds, matching other relays.

## Venue 2: Velodrome (Optimism)

### Overview
- Dominant DEX on Optimism (third-largest Ethereum L2)
- Same team and codebase as Aerodrome
- $5.6B chain TVL
- AMM model: identical to Aerodrome (ve(3,3) + concentrated liquidity)
- Venue type: amm

### Key Point
Velodrome and Aerodrome are built by the same team with nearly identical
smart contracts. The Velodrome relay should be a clone of the Aerodrome
relay with only the following changes:
- RPC endpoint: Optimism RPC instead of Base RPC
- Token addresses: Optimism token addresses instead of Base addresses
- Venue identifier: "velodrome" instead of "aerodrome"
- LIS ref prefix: "lis-velodrome-" instead of "lis-aerodrome-"

### Data Access
Same options as Aerodrome:
- Expand Network REST API (use Velodrome DexId - check docs)
- Bitquery GraphQL with Optimism chain filter
- Direct on-chain reads via Optimism RPC

## Venue 3: PancakeSwap (BNB Chain)

### Overview
- Dominant DEX on BNB Chain
- $903M 24h volume (12.5% of global DEX volume)
- AMM model: Uniswap V3 fork (concentrated liquidity)
- Venue type: amm

### Key Point
PancakeSwap V3 is a Uniswap V3 fork. The depth and spread calculation
logic from the existing uniswap-relay.ts can be reused directly. The
main differences are the RPC endpoint (BNB Chain), token addresses
(BEP-20 format), and venue identifier.

### Data Access (in order of preference)

**Option A: PancakeSwap Subgraph API (recommended)**

    first: 5

    id
    token0 { symbol }
    token1 { symbol }
    totalValueLockedUSD
    tick
    sqrtPrice
    liquidity
    feeTier

**Option B: Expand Network REST API**
Same expand.network service as Aerodrome.

**Option C: Direct on-chain reads via BNB RPC**
