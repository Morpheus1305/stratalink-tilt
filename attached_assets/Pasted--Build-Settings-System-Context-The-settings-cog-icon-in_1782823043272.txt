# Build Settings System

## Context

The settings cog icon in the header bar (`dashboard-header.tsx`) is currently non-functional. It needs to become a comprehensive settings interface that controls display preferences, supervised portfolio configuration, notification preferences, report delivery, session management, and system status. Settings must be persisted in the database per user and applied consistently across every page in the platform.

---

## Settings Panel Design

When the user clicks the settings cog, a slide-out panel appears from the right side (same interaction pattern as the notification bell and reports panel). The panel has six sections accessible via a vertical navigation list on the left side of the panel, with the content area on the right.

### Panel styling

- Width: 480px
- Background: `#0B1019` (bg-primary)
- Left border: 1px `#1A2435`
- Slide-in animation: 0.2s ease-out from right
- Close: X button, click outside, or Escape key
- Header: "SETTINGS" in 12px monospace, `#00BFA5`, with close button
- Section navigation: Vertical list on the left (120px wide), 11px monospace, `#7B8EA3`, active section highlighted with `#00BFA5` text and a 2px left accent bar
- Content area: Right side, scrollable
- Save button: Fixed at the bottom of the panel. "SAVE CHANGES" in 11px monospace, `#0B1019` text on `#00BFA5` background, 2px radius. Only enabled when changes have been made.
- All inputs follow house style: `#0F151F` background, 1px `#1A2435` border, 2px radius, `#D8DEE8` text, monospace for data inputs

## Section 1: Display Preferences

### Timezone

```
TIMEZONE
[UTC+4 Abu Dhabi, Dubai (Gulf Standard Time)  ▾]
All timestamps across the platform will display in this timezone.
```

- Dropdown selector with common timezones, defaulting to UTC
- Pre-populated options should include: UTC, UTC+1 (London/BST), UTC+4 (Abu Dhabi/GST), UTC+8 (Singapore/HKT), UTC-5 (New York/EST), UTC-4 (New York/EDT)
- The selected timezone must be applied to every timestamp displayed anywhere in the platform:
  - TILT terminal: "LAST UPDATE" timestamp
  - STRATA AI: Intelligence signal timestamps
  - PoLi/PoMI: Snapshot timestamps
  - RCL: Verified At, Valid Until, Last Event timestamps in Provenance cards
  - Alerts page: Alert timestamps
  - Notification bell: Alert timestamps and relative time calculations ("2 min ago")
  - Reports panel: Report history timestamps
  - ILU-20 ticker: If timestamps are shown
  - Generated reports: All timestamps in PDFs
  - Page footer: Clock display

### Number Format

```
NUMBER FORMAT
Decimal separator:  [. (Period)]  ▾
Thousands separator: [, (Comma)]  ▾

Preview: $14,200,000.00  |  0.08%  |  1.4 bps
```

- Two dropdown selectors for decimal and thousands separators
- Options: Period/Comma, Comma/Period (European), Comma/Space
- Applied to all numeric values across the platform (depths, scores, percentages, spreads)

### Default Token


- Dropdown showing the ILU-20 tokens (same grouped format as the main token selector)

- For now, only "Dark" is available. Include the selector as a placeholder for future light/high-contrast themes.

- Checkboxes grouped by ILU category
- Each token shows symbol and name
- Tokens with active venue data show a green dot. Tokens without show a grey dot.
- Counter at the bottom showing "X of 20" selected
- "Select All" and "Clear All" buttons

### Where the supervised portfolio is consumed

This setting must feed into every relevant part of the platform:

1. **Scheduled reports (Daily, Weekly, Monthly):** Cover only the supervised tokens. The report generation service reads the supervised portfolio from the user's settings.

2. **RCL view:** The Coverage panel should reference the supervised portfolio. "Supervised Instruments: 5" should match the count set here.

3. **STRATA AI page:** The ILU-20 portfolio section should highlight which tokens are in the supervised set (e.g., with a subtle border or marker).

4. **Alerts:** Alert rules can be configured globally or per supervised token. The supervised portfolio defines the default scope for alert monitoring.

5. **Notification bell:** By default, only show notifications for supervised tokens. Include an option (in Notification Preferences below) to see alerts for all ILU-20 tokens.

### Database storage

```sql
CREATE TABLE user_settings (
  user_id INTEGER PRIMARY KEY REFERENCES users(id),
  timezone VARCHAR(50) DEFAULT 'UTC',
  decimal_separator VARCHAR(5) DEFAULT '.',
  thousands_separator VARCHAR(5) DEFAULT ',',
  default_token VARCHAR(10) DEFAULT 'BTC',
  supervised_tokens TEXT[] DEFAULT ARRAY['BTC','ETH','SOL','XRP','AVAX'],
  notification_severities TEXT[] DEFAULT ARRAY['HIGH','MEDIUM','LOW'],
  notification_email VARCHAR(255),
  notification_email_enabled BOOLEAN DEFAULT FALSE,
  notification_scope VARCHAR(20) DEFAULT 'supervised',
  report_email VARCHAR(255),
  report_email_enabled BOOLEAN DEFAULT FALSE,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## Global Integration: Settings Context

### Create a Settings React Context

Similar to the Token Context, create a Settings Context that provides user settings to every component in the platform.

Create `client/src/contexts/SettingsContext.tsx`:

```typescript
interface UserSettings {
  timezone: string;
  decimalSeparator: string;
  thousandsSeparator: string;
  defaultToken: string;
  supervisedTokens: string[];
  notificationSeverities: string[];
  notificationScope: 'supervised' | 'all';
  notificationEmail: string | null;
  notificationEmailEnabled: boolean;
  reportEmail: string | null;
  reportEmailEnabled: boolean;
}
```

Wrap the application in `SettingsProvider` at the top level (alongside `TokenProvider`).

### Where settings are consumed

Every component that displays a timestamp must read the timezone from settings and format accordingly.

Every component that displays a number must read the decimal/thousands separators and format accordingly.

Create two utility functions available globally:

```typescript
function formatTimestamp(date: Date, settings: UserSettings): string
function formatNumber(value: number, settings: UserSettings, options?: { decimals?: number, prefix?: string, suffix?: string }): string
```

These functions are called everywhere a timestamp or number is displayed. This ensures consistency. Do not format timestamps or numbers inline. Always use the utility functions.

### Pages that consume specific settings

| Setting | Consumed By |
|---------|------------|
| Timezone | Every page (all timestamps), reports, notifications, alerts |
| Number format | Every page (all numeric values), reports |
| Default token | App.tsx (set token context on login) |
| Supervised portfolio | Scheduled report generation, RCL supervised count, STRATA AI portfolio highlight, alert default scope, notification filtering |
| Notification severities | Notification bell badge count and panel filtering |
| Notification scope | Notification bell token filtering |
| Notification email | Alert service email delivery |
| Report email | Report generation service email delivery |
| Report schedule | Report generation cron jobs |

### Loading settings on login

When the user signs in:
1. Fetch their settings from `GET /api/settings`
2. Populate the Settings Context
3. Set the Token Context to the user's default token
4. The rest of the application renders with the correct timezone, format, and portfolio

If no settings exist for the user (first login), create a default settings record with sensible defaults (UTC timezone, period decimal, BTC default token, empty supervised portfolio).

## API Endpoints

```
GET    /api/settings              - Returns current user settings
PUT    /api/settings              - Updates user settings (full or partial)
GET    /api/system/status         - Returns system status (read-only)
POST   /api/settings/password     - Change password (current + new)
POST   /api/settings/sign-out-all - Invalidate all sessions
```

All endpoints require authentication.

## What NOT to Change

- Do not modify any data computation logic (L5F, PoLi, STRATA AI, TSLE buffer).
- Do not modify venue relay files.
- Do not modify the existing page layouts or component structures.
- Only add the settings panel, the settings context, the formatting utility functions, and the integration points where settings are consumed.
