# Phase 2: Connect Security Token Exchange Venues

## Context
Phase 1 added RWA tokens (BUIDL, OUSG, PAXG, XAUT, ONDO) to the
monitored asset universe using existing venue connections. Phase 2
extends the venue infrastructure to include dedicated security token
exchanges and tokenization platforms - the venues where digital
securities primarily trade.

Run this prompt AFTER Phase 1 is live and verified.

## Why This Matters
Digital securities (tokenized equities, bonds, funds, commodities) do
not trade on Binance or Uniswap. They trade on regulated security
token exchanges that operate as Multilateral Trading Facilities (MTFs)
or Alternative Trading Systems (ATS). These venues are regulated,
KYC-gated, and operate under securities law rather than virtual asset
rules.

For ADGM, this is the bridge between their virtual asset framework and
their securities framework. Showing liquidity verification across both
venue types demonstrates that Stratalink is a regulatory infrastructure
layer, not a crypto monitoring tool.

## Important Architectural Difference

Security token exchanges differ from DEXs and CEXs in several ways
that affect relay design:

| Factor | CEX/DEX (existing) | Security Token Exchange (new) |
|--------|-------------------|-------------------------------|
| Access | Public APIs | KYC-gated APIs, may require agreements |
| Order books | Continuous, public | May be RFQ, auction, or continuous |
| Settlement | T+0 (on-chain) | T+0 to T+2, may involve off-chain steps |
| Data format | REST/WebSocket/subgraph | Varies - REST, FIX protocol, proprietary |
| Token standard | ERC-20 | ERC-20, ERC-1400 (security token), ERC-3643 |
| Compliance | Minimal | Embedded (whitelists, transfer restrictions) |

The relay for these venues may need to handle:
- API key authentication (not public endpoints)
- Different data formats (not standard order book depth)
- Lower update frequency (these markets move slowly)
- Compliance-restricted data (not all data may be public)

## New Venue Type: Regulated Security Token Exchange

Add a new venue type to shared/venue-config.ts alongside existing types
(rest, relay, amm, rfq):

```typescript
type VenueType = "rest" | "relay" | "amm" | "rfq" | "ats" | "mtf";
```

- **ats**: Alternative Trading System (US-regulated)
- **mtf**: Multilateral Trading Facility (UK/EU/international regulated)

These venue types should render differently in the UI:
- RCL Provenance panel: show "ats" or "mtf" label instead of "rest" or "amm"
- Heatmap: these venues are regulated by definition, so the regulatory
  bonus in the stability score applies
- Venue Depth & Spread Monitor: may show different data fields
  (AUM instead of depth for fund products)

## Implementation Approach

### Phase 2a: Securitize + Archax (immediate value)
Start with Securitize (BUIDL primary market data) and Archax
(FCA-regulated venue). These two provide the most strategic value:
- Securitize gives primary market depth for BlackRock BUIDL
- Archax gives FCA-regulated digital securities coverage
Together they bridge ADGM's virtual asset and digital securities
frameworks.

### Phase 2b: INX + tZERO + SDX + ADDX (broader coverage)
Add the remaining venues for comprehensive security token exchange
coverage across US, Swiss and Singapore regulatory jurisdictions.

### API Access Considerations
Unlike DEX and CEX APIs, security token exchange APIs typically
require:
- Formal onboarding / account creation
- API key provisioning
- Possibly data licensing agreements
- KYC/institutional verification

This means Phase 2 has a business development component alongside the
technical build. Rob may need to reach out to each platform's
partnership or API team to establish access. The relay code can be
written against their documented API specifications, but live data
flow requires an authenticated connection.
