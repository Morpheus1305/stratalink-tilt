# Phase 2: Add Venue Relays - SyncSwap (zkSync), Linea DEX, Scroll DEX, Uniswap V3 (World Chain)

## Context
Phase 2 completes EVM L2 coverage by adding the four remaining
EVM-compatible L2s with TVL above $1B. Run this prompt AFTER Phase 1
(Aerodrome, Velodrome, PancakeSwap) is live and verified. The relay
patterns from Phase 1 are directly reusable here.

After Phase 2, Stratalink will cover every EVM-compatible L2 with TVL
above $1 billion - 20 venues across 10 chains, representing 97% of all
EVM L2 DeFi TVL.

## Venue 1: Uniswap V3 (World Chain)

### Overview
- World Chain is an OP Stack L2 ($1.8B TVL)
- Uniswap V3 is deployed on World Chain
- Venue type: amm

### Key Point
This is the easiest venue in the entire expansion. Uniswap V3 is
already connected on Ethereum mainnet. The World Chain relay is an
exact clone of the existing uniswap-relay.ts with only the RPC endpoint
and token addresses changed. The contract ABIs, pool discovery logic,
depth calculation and spread derivation are all identical.

### Data Access
- Direct on-chain reads via World Chain RPC (preferred - same contract
  interfaces as Ethereum Uniswap V3)
- Expand Network or subgraph if available for World Chain

### Relay Implementation
Create: server/services/uniswap-worldchain-relay.ts

Clone uniswap-relay.ts and update:
- RPC endpoint to World Chain
- Token addresses to World Chain equivalents
- Venue name to "uniswap-worldchain"
- Chain identifier to "worldchain"
- LIS ref prefix to "lis-uniswap-worldchain-"

### Token Addresses
Look up ILU-20 token addresses on World Chain at time of implementation.
WETH and major stablecoins (USDC, USDT) will be available. Coverage for
smaller ILU-20 tokens may be limited - return null for unlisted tokens.

## Venue 2: SyncSwap (zkSync Era)

### Overview
- Dominant DEX on zkSync Era ($4.1B chain TVL)
- zkSync Era is a zkEVM - uses zero-knowledge proofs for settlement
  but is fully EVM-compatible at the application layer
- AMM model: classic AMM + stable pools (similar to Curve)
- Venue type: amm

### Data Access

**Option A: SyncSwap Subgraph/API**
- Check for SyncSwap's public subgraph or API at time of implementation
- GraphQL query pattern similar to PancakeSwap/Uniswap subgraphs

**Option B: Direct on-chain reads via zkSync RPC**
- zkSync Era RPC: https://mainnet.era.zksync.io
- SyncSwap Router: check SyncSwap docs for current contract addresses
- Standard EVM contract reads work on zkSync Era

**Option C: Expand Network**
- Check expand.network for zkSync Era / SyncSwap support

### Relay Implementation
Create: server/services/syncsswap-relay.ts

1. **Pool discovery**: SyncSwap uses a pool factory pattern. For each
   ILU-20 token, find the primary pool (TOKEN/USDC or TOKEN/WETH).
   SyncSwap has two pool types: Classic Pools (constant product, like
   Uniswap V2) and Stable Pools (optimised for like-valued assets,
   like Curve). Use Classic Pools for volatile assets, Stable Pools
   for stablecoin pairs.

2. **Depth derivation**: For Classic Pools, use the constant-product
   AMM formula (same as Uniswap V2 depth calculation). For Stable
   Pools, use the StableSwap invariant (same as Curve depth
   calculation). Both patterns already exist in the codebase.

3. **Spread derivation**: Derived from pool fee and liquidity depth.
   SyncSwap fees are typically 0.1% for stable pools and 0.3% for
   volatile pools.

4. **Normalisation output**:
   ```typescript
   {
     venue: "syncswap",
     venueType: "amm",
     chain: "zksync",
     depth: number,
     spread: number,
     lastEvent: number,
     lisRef: `lis-syncswap-${timestamp}`
   }
   ```

5. **Polling**: Every 5 seconds.

- Multiple competing DEXs - select highest-TVL at time of connection
  (candidates: Ambient, Nuri, Uniswap V3 on Scroll)
- Venue type: amm

### Key Point

Scroll, the existing Uniswap relay logic works without ANY modification
except the RPC endpoint and token addresses. Check whether Uniswap V3
is live on Scroll at time of implementation - if so, this is another
zero-effort clone like World Chain.
